## 1. 直すべき問題

3回目の点検（2026-07-03）で見つかったもの。優先度順。

### 1.1 【中】スマホ幅でスコア入力画面の右側が切れて操作できない（作業中のスマホ対応の不具合）

- 現象: 幅390pxで開くと、レイアウト用グリッドの1列トラックが698pxまで広がり、スコアボードの右半分（得点・ベース・カウント）と各ボタン行の右端が画面外に切れる。`main` に `overflowX: hidden` が指定されているため、横スクロールでも到達できず操作不能。
- 場所: `src/client/pages/ScoreInputPage.jsx` の `gridTemplateColumns: { xs: "1fr", ... }`（未コミットのスマホ対応変更に含まれる）。
- 原因: CSS Gridの `1fr` は `minmax(auto, 1fr)` として振る舞い、グリッド子要素の自動最小サイズがトラックを押し広げる既知のパターン。
- 解決策（実証済み）: xs側のトラックを `"minmax(0, 1fr)"` に変える。実ブラウザで適用したところ、scrollWidth 714px→390px となり、スコアボードが374pxに収まって全ボタンが表示された。1行の修正。

### 1.2 【中】自己ホストWebフォントが未実装で、端末によってスコアボードの文字が変わる

- 現象: `rules.md` の技術スタック、ファイル構成の `src/client/fonts/`、`scoreboard_design.md` 3.3、`theme.js` のコメントはいずれも「自己ホストのWebフォントを同梱」とするが、`fonts/` フォルダ自体が存在しない。スコアボードは `Bahnschrift`（Windows固有）を先頭にしたフォールバック頼みのため、スマホやLinuxではOS標準フォントで描画され、端末ごとに見た目が変わる。
- 影響: 技術選定の最優先事項「配信で見劣りしない、必要十分にリッチなスコアボード表示」に対して、配信PC以外で見た目の一貫性が保てない。
- 解決策: 再配布可能なコンデンスド・ゴシック（例: Saira Condensed / Barlow Condensed）と Noto Sans JP サブセットの woff2 を `src/client/fonts/` に同梱し、`@font-face` を定義してテーマとスコアボードSVGの `<style>` から参照する。フォント選定＋適用でやや作業量あり。

### 1.3 【小】data_model.md §6 に実装に存在しないフィールドが残っている

- 現象: Viewer Settings の保持情報に `viewer client id` と `board layout mode` が挙がっているが、実装（`viewer/viewerSettings.js`）には存在しない。
- 解決策: 実装に合わせて2行を削除するか、将来項目である旨を明記する。

### 1.4 【小】ファイル構成にある `src/shared/types.js` が未作成

- 現象: `rules.md` のファイル構成に「JSDoc typedef（@ts-check で共有する型の形）」として記載があるが、ファイルが存在しない（`src/shared/` には `scoringRules.mjs` のみ）。
- 解決策: 共有typedefを実際に作るか、ツリーから外す。`fonts/` の欠落は1.2で解消される。

## 2. 直さなくていい問題

理由と、再検討する条件を記す。

### 2.1 プリセット保存が「プリセット単位の全項目・後勝ち」であること

- 指摘: Settings Pageの `savePreset` は全項目をまとめて送るため、保存直前に他端末が同じプリセットを更新していると後勝ちで上書きされる。
- 判断: 直さない。`operation.md` の「複数端末で編集した場合、最後に保存した内容を有効とする（後勝ち）」という仕様どおりの動作。競合が起きやすい編集メニュー（`board:patchConfig`）と選手名メニュー（`players:patch`）は既に差分パッチ化済みで、残るプリセット編集は単独の管理者が設定画面で行う前提のため、衝突の現実性が低い。
- 再検討する条件: 複数人が同時にプリセットを管理する運用が実際に始まったら、`presets:patch`（フィールド単位の差分保存）を追加する。

### 2.2 `players:patch` が変更ゼロでも `changed: true` を返す

- 指摘: 空のパッチでも `updatedAt` が更新され、SSEが1回配信される。
- 判断: 直さない。実害は再描画1回で、体感差はない。他の修正でこの周辺を触る機会があれば、`board:patchConfig` と同様の変更検知を入れる程度でよい。

### 2.3 SSEにキープアライブ（定期ping）がない

- 判断: 直さない。LAN内・数端末の直結構成では中間プロキシによる切断が起きず、切れてもEventSourceが自動再接続して `connected` イベントで全量同期する。
- 再検討する条件: リバースプロキシ越しの配信や、接続が頻繁に切れる症状が出たら。

### 2.4 Viewerのドラッグ中、pointermoveごとにlocalStorageへ書き込む

- 判断: 直さない。実測で動作は正常。書き込みは同期だがデータは小さく、対象端末（配信PC・スマホ）で体感の引っかかりは確認されていない。
- 再検討する条件: 低スペック端末でドラッグが目に見えてカクつく報告が出たら、ドラッグ終了時のみ保存に変える。

### 2.5 設定PATCHで送られた未知のキーがそのまま保存される

- 指摘: サーバーの `/api/settings` が `...body` で全キーを取り込むため、想定外のキーがapp.jsonに残り得る。
- 判断: 直さない。単独利用のローカルアプリで攻撃面がなく、既知キーは個別に正規化済み。気になる場合は許可キーだけ抽出する3行の修正だが、優先度は最も低い。

### 2.6 スコア入力の各ボタンがサーバー応答まで全体ロックされる

- 指摘: 操作のたびに `pending` で全ボタンが無効化されるため、超高速の連打はできない。
- 判断: 直さない。LANの応答は数ms〜数十msで体感できず、二重送信防止のメリットの方が大きい。

### 2.7 盗塁死用の `keepCurrentBatterOnNextInning` フラグが書き込まれるだけで読まれていない

- 指摘: `scoringRules.mjs` で盗塁死時にtrue、チェンジ時にfalseを設定するが、参照箇所がない（打者はチェンジで進まない仕様のため、フラグなしでも「盗塁死でも現在の打者を維持」は自然に満たされる）。
- 判断: 急いで直さない。実害はなく、`rules.md` の Open Questions で管理済み。次にこの周辺を触るときに削除する。

### 2.8 ページ切り替え時に一瞬白画面になることがある

- 指摘: コード分割（React.lazy）導入により、初めて開くページはチャンク取得までの間 `Suspense fallback={null}` で空白になる。
- 判断: 直さない。LANではチャンク取得が数ms〜数十msで知覚困難。気になる場合はfallbackにローディング表示を入れる程度でよい。

## 3. 解決済みの記録

### 2026-07-02（1回目の点検）で解決を確認したもの

- 設定画面の全体設定フォームが、SSE更新（他画面のスコア操作など）のたびに入力中の値を失う問題 → dirty管理で修正済みを確認。
- 編集メニューの「略称拡大率(%)」数値入力が、入力のたびに即クランプされて実質入力不能だった問題 → blur/Enter時クランプへ修正済みを確認。
- 設定画面のプリセット「ロゴ削除」が選択済み未保存画像（pendingLogo）を消さない問題 → `clearPresetLogo` で両方クリアするよう修正済みを確認。
- 選手名メニューが開いた時点のスナップショットを丸ごと保存し、未編集の球数・成績まで巻き戻す問題 → 編集した項目だけを送る `players:patch` へ改善済みを確認。
- 一時演出がSSE再描画に依存し、期限切れ後も残ることがある問題 → クライアント側タイマーで期限時に消えるよう修正済みを確認。
- スコアリセット機能は `operation.md`（確認ダイアログ必須・リセット対象・履歴クリア）に文書化済みで、実装と一致することを確認。

- 1.1 チャンク分割: `src/client/main.jsx` を `React.lazy()` + `<Suspense>` に変更。ビルド結果は単一603.30kBから、最大チャンク328kB（`index-*.js`）＋ページ別チャンク（例: `ScoreInputPage` 40.89kB、`SettingsPage` 9.86kB）に分割され、500kB超の警告が解消したことをビルド出力で確認。

- 1.4 リクエストサイズ上限: `src/server/index.mjs` の `readJsonBody` に2MB上限を追加。超過時は413を返す（`statusCode` 付きErrorをthrowし、トップレベルの`catch`で処理する経路を新設）。3MBのボディを送って413が返ることを実測確認。

全項目: `npm test` 合格、`npx vite build` 成功、コンソールエラーなしを確認済み。
