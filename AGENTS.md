# AGENTS.md

このファイルは、このリポジトリ（野球スコアボード配信アプリ）で作業する AI Coding エージェント共通のプロジェクト指示ファイルである。
特定エージェントだけに適用する内容は、必ず「エージェント別補足」に分けて明示する。

## プロジェクト概要

野球のスコアボードを LAN 内の配信 PC・スマホ数台で表示するアプリ（OoAIBoard_Baseball）。
ユーザーはプログラマではないため、わかりやすい起動方法、壊れにくい操作、設計書との整合性を重視する。

技術スタックや実装状況など、プロジェクト全体の決めごとは `docs/rules.md` を正とする。
作業前に `docs/rules.md` と、作業内容に関係する設計ファイルを読むこと。

各エージェントは、自分に明示された専用ルールだけを適用する。
他エージェント専用と書かれたルールは適用しない。

## 共通ルール

### 1. Git

- 作業がまとまり、テストに成功するたびにローカル git へコミットする方針を基本とする。ただし、ユーザーが git 保存を求めていない小さな確認・調査では、勝手にコミットしない。
- 「ローカル git に反映して」「手動変更を含めて git 保存して」と指示された場合は、手動編集されたファイルも含めてコミット対象に含める。ただし `.gitignore` の設定を優先し、ignore 対象に手動変更ファイルを追加しない。
- 絶対にリモートへ push しない。別途明示的な指示がない限り厳守する。
- コミット前に、必要に応じて該当する実装リスト（`docs/rules.md` §2.2）や docs を更新し、可能な範囲で `npm test` と `npx vite build` を通す。
- Git 操作が失敗した場合は、変更内容と手動コミット用コマンドを報告する。
- 作業開始時またはコミット前に、自分が触っていない手動変更が見えた場合は勝手に戻さない。関連する場合は内容を読んで合わせ、無関係ならそのまま残す。
- `dist/` と `storage/` は通常 Git 管理対象外の実行時・ビルド生成物として扱う。ユーザーから明示指示がない限り、コミット対象にしない。

### 2. 進め方

- 実装前に既存ファイル構成を確認する。
- 仕様書や設計書がある場合は、それを正として扱う。
- 大きな変更は小さい手順に分ける。
- ユーザーが「設計から」「コードにはまだ入らないで」と明示した場合は、設計・提案に留め、コード本体を編集しない。
- `docs/rules.md` §2.2 の実装リストを進める場合は、指示された手順だけを 1 手順ずつ実行する。各手順ごとに、完了、テスト、必要な設計ファイル更新、コミットのサイクルを回す。
- ユーザーは技術選定を細かく判断できないことがあるため、判断材料が揃っている場合は AI エージェント側で比較検討して決定し、理由を設計ファイルに記録する。
- ファイル追加や大きな分割を行う場合は、`docs/rules.md` の File Structure に沿う。置き場所が不明な説明ファイルや補助ファイルを増やすより、既存の docs / src / tests / storage の役割に合わせる。

### 3. テスト・検証

- 各実装ステップ後に、可能な範囲でテストまたは動作確認を行う。
- 「実画面テスト」を指示された場合は、PC 幅とスマホ幅の両方を画面監視付きで確認する。表示崩れ、操作不能、横スクロールの有無、主要ボタン操作、コンソールエラーを確認する。
- 検証で実データ（ボードやプレイヤー成績など）を操作した場合は、終了前に可能な限り元の状態へ復元する。
- 表示位置、文字位置、重なり、スクロール挙動などの修正は、推測で済ませず実ブラウザで確認してから報告する。
- 実画面テストで一時的にローカルサーバーを起動した場合は、確認後に停止する。確認用ログを作った場合は、最後に削除して作業ツリーへ残さない。
- `tests/` ディレクトリは原則読み取り専用。変更するのは、ユーザー指示に基づく相応の理由がある場合のみ。

### 4. 問題点の記録

問題点を収集して `docs/problems.md` へまとめる指示があった場合は、以下の通り実行する。

- 設計ファイルと実装の突き合わせ、コード通読、実ブラウザでの動作確認を行う。
- 見つかった問題は「1. 直すべき問題」（優先度順）と「2. 直さなくていい問題」（判断理由と再検討条件を明記）に分類する。
- 対処が完了したら「3. 解決済みの記録」に、確認日・確認内容とともに移す。
- 問題が見つからない場合、ユーザーが「正常ならファイルは変更しない」と指示しているときは `docs/problems.md` を編集しない。

### 5. コミュニケーション

- 返答は日本語で行う。
- ユーザーには非技術者にも分かる言葉で説明する。必要な場合だけ、ファイル名・コマンド名・技術用語を補足する。
- 指示がピクセル数・パーセンテージ単位など具体的な場合は、その条件を優先して確認する。
- 曖昧な点があっても、ローカルの設計書やコードから合理的に判断できる場合は自分で進める。判断が危険な場合だけ、短く確認する。

## エージェント別補足

### Codex のみ

- `docs/rules.md` 内に Codex 以外のエージェント専用ルールがある場合、Codex はその節を適用しない。例: `docs/rules.md` §3.1 は Codex には適用しない。
- Codex が実画面テストで確認用ログを作る場合は、`codex-server.out.log` / `codex-server.err.log` のような一時ファイル名を使い、作業完了前に削除する。

## 主要ドキュメント

- `docs/rules.md`: 技術スタック、実装順、コーディング規約、未決事項。
- `docs/scoreboard_design.md`: 画面構成、スコアボードの見た目、メニュー。
- `docs/operation.md`: 操作時の状態変化、ルール、各操作の意味。
- `docs/data_model.md`: データ構造。
- `docs/button_list.md`: 必要なボタン一覧。
- `docs/problems.md`: 発見済みの問題点と解決済み記録。
- `Readme.md`: 起動、終了、基本的な使い方。
